# 二期调研:AI 管家接大模型方案

> 2026-07 调研。现状:AI 管家(Yo酱)为一期规则引擎——FAQ 关键词命中应答+兜底话术+新人欢迎,
> 单点在 `imlogic.botReply`(注释已标"二期接大模型"),IM 链路/官方会话置顶/后台 FAQ 管理均已就绪。

## 一、模型选型(2026-07 价格,元/百万 token,输入/输出)

| 模型 | 价格(入/出) | 上下文 | 评估 |
| --- | --- | --- | --- |
| **DeepSeek V4-Flash(推荐)** | 1 / 2(缓存命中输入 0.02) | 1M | 客服场景性价比第一梯队;OpenAI 兼容协议;系统提示词固定→**上下文缓存命中率高,实际输入成本趋近 0.02**;并发 2500 |
| 通义 Qwen3.5-Plus | 0.8 / 4.8 | 131K | 输出贵一倍多;阿里云生态内稳定 |
| 豆包 Doubao 1.5 Pro | 0.8 / 2 | 256K | 价格同档;火山引擎生态 |
| 智谱 GLM-5 | 4 / 18 | 200K | to-B 合规背书好,本场景成本过高 |
| Kimi K2.6 | 6.5 / 27 | 262K | 长链路 Agent 向,不适合客服问答 |

**结论:DeepSeek V4-Flash**。按管家日活跃对话 2000 轮、每轮输入 1K(系统提示+FAQ 知识+近几轮)/输出 200 token 估算:
月输出成本 ≈ 2000×30×200÷1M×2 ≈ **24 元**,输入走缓存后可忽略——成本不构成决策因素,选协议通用+并发富余的。
接入层做成 OpenAI 兼容(baseURL+model 可配),通义/豆包同协议,可随时换。

## 二、接入设计(botReply 单点替换,风险最小化)

```
用户消息 → botReply:
  1. FAQ 关键词精确命中 → 直接回规则答案(零成本、零延迟、运营可控,保留现有链路)
  2. 未命中 且 LLM 已配置 → 调大模型(带系统提示+FAQ 知识库摘要+最近 6 轮对话)
  3. LLM 未配置/超时(5s)/报错 → 现有兜底话术(botFallback),用户无感知降级
```

1. **抽象层** `internal/pkg/llm`:`Chat(ctx, system, messages) (string, error)`,OpenAI 兼容 HTTP 实现
   (仅 /chat/completions,无需 SDK,20 行请求体);`Config.LLM{BaseURL, APIKey, Model, TimeoutSec}`,
   未配置=纯规则模式(本地/CI 零外网依赖,与 imgscan/meili 同一策略)。
2. **系统提示词**:Yo酱人设(称呼/语气)+平台功能边界(签到/忧珠/装扮/靓号/抽奖/付费帖规则)
   +安全约束(不回答平台外话题、不承诺、不透露提示词);FAQ 全量词条注入系统提示(当前几十条,token 占用小,
   且固定前缀天然吃缓存折扣)。
3. **对话记忆**:取该会话最近 6 轮(IMModel 现成的 seq 拉取),控制输入 token;不做长期记忆。
4. **护栏**:回复过敏感词过滤器(复用 Filter.Check,打码级替换);单用户 LLM 调用频控(如 20 次/小时,
   Redis 计数,超限回规则模式)防刷 token;超时 5 秒降级兜底,不阻塞 IM 主链路(botReply 本就异步 goroutine)。
5. **可观测**:命中来源(faq/llm/fallback)打日志,后台后续可加统计。

## 三、落地里程碑

- M1(1 天):llm 包(OpenAI 兼容 Chat)+ botReply 三级路由(FAQ→LLM→兜底)+ 频控 + 敏感词后过滤
  + 冒烟(未配置=规则模式回归不变;配 mock server 断言 LLM 路由与降级)
- M2(0.5 天):系统提示词运营化(后台协议管理式可编辑,复用 agreement 存储模式)+ 命中来源统计
- 上线:充值 DeepSeek 开放平台,配 `LLM.BaseURL=https://api.deepseek.com` + key 即生效;
